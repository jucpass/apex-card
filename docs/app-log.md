### Members row-action bug fixes
- Fixed a race condition in `MemberEditDialog`: the Edit dialog opened immediately while member details were still being fetched, with no loading state. Submitting before the fetch resolved silently no-op'd (the dialog's still-mounted overlay then blocked all further clicks anywhere on the page, reading as "row actions do nothing"). Fixed by showing a loading skeleton until the member's data has actually arrived, so the form/Save button can't be interacted with prematurely.

- **Root cause of a second, more fundamental bug** ("the '...' actions dropdown never opens"): `apps/admin-web/src/components/ui/button.tsx`'s `Button` was a plain function component, not wrapped in `React.forwardRef`. When used as the `render` target for base-ui `DropdownMenuTrigger`/`DialogTrigger`/etc. (as in `MemberRowActions`), the ref that Floating UI needs to measure the trigger's position was silently dropped, so popups opened (correct ARIA state, zero console errors) but rendered anchored at the viewport's top-left corner (0,0) instead of near the button â€” invisible to a real user, indistinguishable from "nothing happens". Fixed by wrapping `Button` in `React.forwardRef` and forwarding the ref to the underlying `ButtonPrimitive`. This is a foundational component used throughout Admin Web, so this fix likely also resolves latent positioning issues for any other popup triggered via a `Button` render target.

### Membership Management simplification: single Membership Type, computed Status, RevenueCat-ready
- **Root cause of the previous round's flaw**: modelling Membership Type and Membership Status as two independent stored fields still allowed nonsensical combinations (`FREE` + `STUDENT`, etc.) â€” nothing in the schema prevented a `FREE`-plan member from also carrying a paid type, and "Granted" was itself stored as a status value, indistinguishable from a real subscription state.
- **New model**: `MembershipType` is now the only concept describing what plan a member is on â€” `FREE | PREMIUM | STUDENT | WORKER` â€” a member has exactly one. Membership Status (`ACTIVE | EXPIRED | CANCELLED`, `null` for `FREE`) is no longer stored at all; it's computed at read time from `type` + `expiresAt` + `cancelledAt`. "Granted" is a separate `isManuallyGranted` boolean describing *how* the current plan was set, not a plan or a status â€” it's always `false` for `FREE` and only ever set alongside a non-`FREE` type by the Grant action.
- **Prisma**: dropped the `MembershipStatus` enum and `Membership.status` column entirely. `MembershipType` gained a `FREE` value (`@default(FREE)`, was `PREMIUM`). Added `Membership.isManuallyGranted Boolean @default(false)`. Two migrations (Postgres requires a new enum value to be committed before it can be referenced): `20260713120000_add_free_membership_type` adds `FREE` to the enum; `20260713120100_simplify_membership_model` adds `isManuallyGranted`, backfills it and `type`/`expiresAt`/`cancelledAt` from every prior `status` value, then drops `status` and the old enum. Verified all 10 seeded members backfilled correctly (including a previously-`EXPIRED` member correctly getting a backfilled past `expiresAt` for status computation) before the column/enum were dropped.
- **`docs/database-schema.md`**: Memberships section rewritten to describe the single-Type model and added a "Membership Status is computed, not stored" subsection with the exact derivation rules and the four admin actions (Grant/Revoke/Reset to Free/Edit).
- **Shared package**: `membershipTypes` now includes `FREE`; added `grantableMembershipTypes` (`PREMIUM | STUDENT | WORKER`, excludes `FREE`) for the Grant/Edit UI; `membershipStatuses` simplified to `ACTIVE | EXPIRED | CANCELLED`. Added `MembershipDuration` (`{ kind: 'unlimited' }` or `{ kind: 'expiryDate'; expiresAt }`) and `MembershipPlanInput` (type + duration), reused by both `GrantMembershipInput` and `UpdateMembershipInput`, with matching zod schemas â€” storing an explicit duration mode (rather than inferring "unlimited" from a null date) keeps the model unambiguous for a future RevenueCat sync, which will need to distinguish "no expiry by admin design" from "no expiry yet because we haven't heard from the store."
- **API**: `services/members.ts` now computes status via `computeMembershipStatus()` and exposes `membershipStatusWhere()` so status filtering compiles to real, indexable Prisma `WHERE` clauses on `type`/`expiresAt`/`cancelledAt` (no raw SQL, no stored-status filter to keep in sync). Added `POST /api/admin/members/:id/membership/reset` (returns a member to `FREE`, clears `isManuallyGranted`/`expiresAt`/`cancelledAt`) alongside the existing grant/revoke/update endpoints, all now taking/returning the new duration-aware shapes.
- **Admin Portal UI**: Members table's Membership column now shows a single Type badge plus, only when relevant, a small muted-text secondary line combining "Granted" and/or a coloured status word (Active=green, Expired=red, Cancelled=grey) â€” Free members show only the Type badge, nothing else, so the invalid "Free + X" combination is now structurally unrepresentable in the UI as well as the data model. Added a shared `MembershipPlanFields` component (Type select + Duration select + conditional expiry-date input) reused by both the Grant dialog and the Edit dialog's membership section, which is hidden entirely for `FREE` members (a "Use Grant membership" hint is shown instead) and otherwise offers change-type/change-duration/switch-duration-mode plus a destructive "Remove membership" action that calls the new reset endpoint. `MemberDetailsSheet` gained a real Membership details section (Type, computed Status, "Granted manually" Yes/No, and a status-contextual date row â€” Renewal/Expiry date if Active, Expired-on if Expired, Cancelled-on if Cancelled).
- Full English/Portuguese translation of all new/changed strings; ran lint, `tsc` typecheck, and production build for `packages/shared`, `packages/database`, `packages/api`, and `apps/admin-web` â€” all clean.
